## 2. 关键选型决策

### 2.1 为什么不用 LangChain Agent？

LangChain 的 message 流转黑盒，无法在 2s SLA 下精准调优。ContextBundle、Prompt Composer、Memory Service 等核心逻辑高度定制，框架会引入不必要的黑盒。LangGraph 是更底层的图编排器，可定制性强。

### 2.2 为什么不用向量数据库？

- 2C4G 跑不动独立向量库 + embedding 模型 + 重排
- 当前记忆规模 < 1 万条，关系型 MySQL + LIKE/全文索引够用
- 预留 MemoryService 端口，未来独立 RAG 服务可平滑接入

### 2.3 为什么不用 Kafka / ClickHouse？

- 2C4G 资源约束
- 日志量不大（日 < 10GB），JSONL + MySQL 索引够用
- 异步任务用 APScheduler 或 cron 替代

### 2.4 为什么 Admin Web 用 Ant Design 而不是 Shadcn？

- 后台管理 AntD 生态更成熟（复杂表格、表单、图表开箱即用）
- 团队熟悉度更高
- 移动端 App 单独走 Flutter 设计系统，详见 [.claude/rules/app-ui.md](../../.claude/rules/app-ui.md)

### 2.5 为什么移动端从 React Native 切到 Flutter？

README 标注 React Native，但实际代码已是 Flutter（`mobile-app/lib/`）。切换原因：
- RN 在复杂动画/手势体验差
- Flutter 一套代码三端（Android/iOS/Web）成本更低
- Dart 类型系统更强
- Flutter 设计上限高（Material/Cupertino 自定义容易）

README 待修订同步。

### 2.6 为什么后端 Python 而不是 Go/Rust？

- AI 生态：Anthropic SDK、LangGraph、LangChain Python 第一手
- 团队熟悉度
- 业务量级（200 农户）Python 性能够用
- 后期如果某个 Skill 是性能瓶颈，可以用 Go/Rust 重写为独立服务

## 3. 依赖管理

| 端 | 工具 | 锁文件 |
| --- | --- | --- |
| Backend | Poetry | `backend/pyproject.toml` + `poetry.lock` |
| Admin Web | pnpm | `admin-web/package.json` + `pnpm-lock.yaml` |
| Mobile | pub | `mobile-app/pubspec.yaml` + `pubspec.lock` |

**禁止规则**：
- 不允许直接 `pip install`、`npm install -g`、`flutter pub add` 修改 lock 之外的依赖
- 升级依赖必须开 PR 说明理由和兼容性
- 安全扫描：后端 `poetry audit`，前端 `pnpm audit`，每月一次

## 4. 外部服务依赖

| 服务 | 用途 | 配置位置 | 失败兜底 |
| --- | --- | --- | --- |
| Anthropic Claude API | 主 LLM | `core/llm_client_manager.py` | 切换国产 Provider |
| 国产 LLM Provider | 兜底 LLM | 同上 | 限流降级 + 错误提示 |
| QWeather | 主天气 | `services/weather_service.py` | 切换 Open-Meteo |
| Open-Meteo | 兜底天气 | 同上 | 提示「天气服务暂时不可用」 |
| LangSmith（可选） | Trace 可视化 | `core/config.py` | 不影响主流程 |
| MySQL | 业务数据 | `core/database.py` | 主备切换（生产） |
| 文件系统 | JSONL event log | `app/observability/` | 不可缺失 |

## 5. 部署拓扑

### 5.1 生产（腾讯云生产服务器）

```
Internet
   │
   ▼
Nginx (443 TLS + 80 redirect)
   │
   ├── / → admin-web 静态资源
   ├── /api → uvicorn :8000
   └── /api/v1/agent/chat/stream → uvicorn (SSE 长连接)
   │
   ▼
uvicorn (FastAPI, Python 3.12)
   │
   ├── MySQL 8.x（本地或远端）
   ├── LLM Provider（公网）
   ├── QWeather / Open-Meteo（公网）
   └── JSONL 文件（本地 <repo_dir>/backend/logs/）

systemd: <service_name>.service（具体名称见部署仓库）
日常运维：见部署仓库 deploy/ 目录
```

### 5.2 开发

```bash
# Backend
cd backend && poetry install
poetry run uvicorn app.main:app --reload

# Admin Web
cd admin-web && pnpm install && pnpm dev

# Mobile
cd mobile-app && flutter pub get && flutter run
```

## 6. 版本治理

| 维度 | 当前版本 | 升级策略 |
| --- | --- | --- |
| Python | 3.12 | 跟随 Ubuntu LTS，不主动升 |
| FastAPI | 最新稳定 | 半年评估一次 |
| LangGraph | 最新稳定 | 跟随官方 |
| Anthropic SDK | 跟随模型 | 模型升级时一起升 |
| React | 19 | 跟随官方 |
| Ant Design | 5.x | 跟随官方 |

1. 在 Evaluation 跑核心 regression suite
2. 对比上一版本通过率、token、延迟
