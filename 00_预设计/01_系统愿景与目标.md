# 01 — 系统愿景与目标

> 状态：草稿 | 维护：BlockShip | 关联：[02_系统功能及技术架构总设计](./02_系统功能及技术架构总设计.md)

---

## 1. 我们在解决什么问题

中国小农户和中小型种植户的核心痛点：

| 痛点 | 表现 |
| --- | --- |
| **记账繁琐** | 一笔账要打开 App、选分类、填金额、选日期，记账成本超过记一笔账本身的价值 |
| **数据散乱** | 成本在 Excel、工人工资在便签、茬口在脑里、天气在朋友圈，无法聚合分析 |
| **不会决策** | 不知道这茬赚不赚、不知道下茬种什么、不知道明天该不该浇水打药 |
| **专家稀缺** | 找农技员贵且慢；通用大模型不懂本地作物、不懂自家历史、不懂赊账 |
| **工具门槛高** | 现有农业 SaaS 学习曲线陡，年龄大、教育程度低的用户用不起来 |

## 2. 我们要成为什么

**做农户真正用得起来的 AI 农场管家**：用对话替代表单、用记忆替代手动、用数据替代直觉、用专家知识替代通用大模型。

不是又一个农业 SaaS Dashboard，也不是又一个 ChatBot。是一个**会记账、会算账、会算天气、会算工资、会算茬口**、且**记得这家农场过去发生了什么**的助手。

## 3. 三年目标

| 维度 | 2026（奠基） | 2027（深耕） | 2028（规模化） |
| --- | --- | --- | --- |
| **用户** | 10 家测试农户稳定使用 | 200+ 农户付费 | 2000+ 农户、跨省覆盖 |
| **作物** | 番茄/辣椒/黄瓜/草莓 4 个模板 | 20+ 主流作物模板 | 用户自定义模板 + AI 推荐模板 |
| **AI** | 单轮工具调用 + 每日建议 + 自然语言记账 | 多轮规划 + Pending 确认 + 跨茬口记忆 | RAG 长时记忆 + 训练微调 + 端云协同 |
| **工程** | 单机部署、LangGraph + Skill | 模块化、Context/Memory/Prompt 工程化 | 多租户、可观测、数据飞轮闭环 |
| **商业** | 验证 PMF | 月 ARR ≥ 5 万 | 月 ARR ≥ 50 万，考虑融资 |

## 4. 设计哲学

### 4.1 对话优先（Conversational First）

凡是能用一句话搞定的，不让用户填表。但要承认表单在「确认/校对/批量录入」场景的不可替代性——所以是 **对话 + Smart Fill 表单双轨**，不是 Only ChatBot。

### 4.2 数据飞轮优先（Data Flywheel First）

每一轮对话都是数据。**真实会话 → 规则候选 → LLM 预标注 → 人工确认 → 回归/评测/训练**这条链路是产品的护城河，不是 nice-to-have。从第一天起就把 trace、event log、debug export、标注队列、simulation 全部串起来。

### 4.3 边界清晰（Boundary First）

Agent Runtime 不写 Prompt；Context Selector 不写 Memory；Skill 不直接读 Memory 表；API 不直接编 Context。**禁止跨边界 import** 是 CI 门禁，不是风格建议。

### 4.4 小服务器优先（Resource-Constrained First）

生产是 2 核 4G。任何架构决策都要回答：「这个方案能不能在 2C4G 上跑？」不允许出现「我们假设以后上 32C64G」的设计。MySQL + JSONL + 异步任务，不引入 Kafka / ClickHouse / 向量库。

### 4.5 可观测优先（Observability First）

每个写操作、每次工具调用、每条 LLM 请求都要有 `trace_id`、`session_id`、`turn_id`、`request_id`。不允许「我不知道这次为什么失败」的情况发生。

## 5. 核心非目标（明确不做什么）

| 不做 | 原因 |
| --- | --- |
| 不做硬件 IoT 集成 | 当前阶段农户感知不到价值，且硬件依赖性强 |
| 不做语音对话 | 留到 2027，先把文本体验打磨好 |
| 不做多模态图片识别（病虫害拍照） | 留到 2027，先解决记账/茬口核心痛点 |
| 不做 SaaS 多租户计费 | 2026 单租户够用，避免过度工程 |
| 不做端云协同推理 | 端侧算力不足，2026 全云推理 |
| 不做自研大模型 | 直接用 Anthropic / 国产 Provider，专注 Agent 工程 |

## 6. 成功指标（North Star）

| 指标 | 2026 目标 | 2027 目标 |
| --- | --- | --- |
| **DAU/MAU** | 30% | 45% |
| **每周对话轮数 / 用户** | ≥ 10 | ≥ 30 |
| **记账 AI 解析成功率** | ≥ 90% | ≥ 95% |
| **每日建议采纳率（点开/执行）** | ≥ 40% | ≥ 60% |
| **写操作 Pending 确认率** | ≥ 80% | ≥ 90% |
| **回归测试通过率** | ≥ 95% | ≥ 98% |
| **DataFlywheel 周均标注样本** | ≥ 50 | ≥ 200 |
| **P95 响应延迟（首 token）** | ≤ 2s | ≤ 1.5s |

## 7. 风险与对策

| 风险 | 优先级 | 对策 |
| --- | --- | --- |
| 大模型成本失控 | P0 | Prompt Caching + 快思维减少 LLM 调用 + 国产模型兜底 |
| 自然语言解析误触发写操作 | P0 | Pending Action 二次确认 + Reflector 一致性检查 + 反思 trace |
| Skill 数量膨胀导致选择不准 | P1 | Skill Coverage Matrix + 触发词去重 + Tool Selector 评测 |
| 长时记忆质量不足 | P1 | 仅记录高置信习惯/偏好 + 人工确认才进真值 |
| 多端体验不一致 | P1 | API 契约 + 共享 Schema + UI 规范统一 |
| 单机宕机数据丢失 | P2 | MySQL 主备 + JSONL 异步备份 + 关键操作幂等 |
