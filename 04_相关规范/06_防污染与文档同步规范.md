## 1. 防污染（避免垃圾文件）

### 1.1 禁止创建

| 文件/目录 | 原因 |
| --- | --- |
| `__pycache__/` | 编译缓存，应由 .gitignore 排除 |
| `.DS_Store` | macOS 系统文件 |
| `*.pyc`、`*.pyo` | 编译缓存 |
| `*.egg-info/` | 打包元数据 |
| `*.tmp`、`*.bak` | 临时文件 |
| `test.py`、`tmp.py`、`debug.py`（根目录） | 临时调试脚本 |
| `.env`（提交） | 含密钥 |
| `node_modules/` | npm 依赖 |
| `venv/`、`.venv/` | Python 虚拟环境 |
| `build/`、`.dart_tool/` | 构建产物 |

### 1.2 禁止写入

- `.git/` 目录内部（用 git 命令操作）
- `venv/`、`node_modules/` 内部
- 用户主目录（除非显式授权）

### 1.3 文件组织

- Python 文件放合理包目录，不在根目录散放
- 临时测试代码放 `backend/tests/`，文件名 `test_*.py`
- 配置文件集中（`backend/config.yaml`、`.env.example`）
- 不创建一次性辅助脚本

### 1.4 代码整洁

- 删除未使用 import
- 删除未使用变量
- 删除被注释掉的代码块
- 不写无意义 docstring（如 `""" Constructor """`）
- 不留 TODO 残留（要么做完，要么开 issue）

## 2. 文档同步

### 2.1 触发条件

| 改动类型 | 必须更新的文档 |
| --- | --- |
| 新增/修改 API 端点 | `docs/reference/api-spec.yaml` + 本 Spec `03_接口协议/` |
| 新增/修改数据模型 | `docs/reference/api-spec.yaml` + 本 Spec `04_相关规范/03_数据库` |
| 新增/修改配置项 | `docs/architecture/overview.md` + 本 Spec `03_接口协议/03_外部服务接口` |
| 新增/修改部署相关 | `docs/architecture/overview.md` + 本 Spec `01_正式设计/07_可观测与运维` |
| 新增/修改安全策略 | `.claude/rules/security.md` + 本 Spec `04_相关规范/02_安全` |
| 新增功能模块 | `docs/design/` 新增设计文档（用 TEMPLATE.md）+ 本 Spec `01_正式设计/` |
| 新增 Skill | 本 Spec `01_正式设计/02_Skill引擎与契约` 第 6 节 Skill 清单 + `docs/agent/skill-coverage-matrix.md` |
| 修改本 Spec 章节 | 同时检查所有交叉引用 |

### 2.2 更新检查

修改代码后，运行：

```bash
bash scripts/check-doc-freshness.sh
```

检查项：
- API spec 与代码一致
- 文档 `last_updated` 字段是否过期
- 是否有未链接的新文档

### 2.3 文档状态标记

每篇文档 frontmatter：

```yaml
---
last_updated: 2026-06-19
status: active  # active | draft | deprecated
---
```

**状态规则**：
- `active`：当前权威
- `draft`：草稿，可能不稳定
- `deprecated`：被替代，不删除（指向新文档）

**禁止**：直接删除被替代的文档，要标 deprecated 并加迁移指引。

### 2.4 PR 自检清单

提 PR 前自检：

- [ ] 代码通过 `ruff check . && ruff format .`
- [ ] 代码通过 `pytest -v`
- [ ] 代码通过 `bash scripts/check-layer-deps.sh`（架构依赖）
- [ ] 代码通过 `bash scripts/check-guide-sensor-pairing.sh`（如适用）
- [ ] 文档已同步（按上表）
- [ ] 没有 console.log / print 调试残留
- [ ] 没有未删除的 TODO
- [ ] 没有 .env / 密钥提交
- [ ] 单文件 ≤ 500 行
- [ ] 新增 Skill 有 skill.md + tests
- [ ] 提交信息符合 Conventional Commits

## 3. Conventional Commits

```
<type>(<scope>): <subject>

<body>

<footer>
```

| type | 用途 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复 |
| `refactor` | 重构 |
| `docs` | 文档 |
| `test` | 测试 |
| `chore` | 杂项 |
| `perf` | 性能 |
| `style` | 格式 |
| `ci` | CI/CD |
| `build` | 构建 |

示例：
```
feat(agent): add create-cost-record skill with pending confirmation

- 实现 create_cost_record Skill 写操作
- 通过 PendingAction 二次确认
- 添加 5 个单元测试 + 2 个集成测试

Closes #123
```

**规则**：
- 中文 commit message 允许，但 type 用英文
- 一个 commit 一个逻辑变更
- 同天多个 commit 合并为一个（squash）

## 4. PR 标题与描述

### 4.1 标题

简洁、动词开头：
- ✅ `feat: 新增记账 Skill`
- ✅ `fix: 修复账单汇总金额单位错误`
- ❌ `update`、`changes`、`misc`

### 4.2 描述

```markdown
## 背景
为什么做这个改动。

## 改动
- 改动 1
- 改动 2

## 测试
- [x] 单元测试通过
- [x] 集成测试通过
- [x] 手工验证场景

## 文档
- 更新 xxx.md
- 更新本 Spec xxx 章节

## 关联
- Closes #123
- Depends on #124
```

## 5. 文件命名约定

| 文件类型 | 规则 | 示例 |
| --- | --- | --- |
| Python | snake_case.py | `cost_service.py` |
| TypeScript | camelCase.ts / PascalCase.tsx | `costService.ts` / `CostList.tsx` |
| Dart | snake_case.dart | `cost_service.dart` |
| Markdown | kebab-case.md 或中文标题 | `skill-engine.md` / `01_Agent平台架构.md` |
| 配置 | kebab-case.yaml | `api-spec.yaml` |
| 测试 | `test_*.py` / `*.test.ts(x)` | `test_cost_service.py` |

## 6. 目录命名约定

| 类型 | 规则 | 示例 |
| --- | --- | --- |
| Python 包 | snake_case | `agent/runtime/` |
| Skill 目录 | kebab-case | `skills/create-cost-record/` |
| 前端组件目录 | PascalCase 或 kebab-case | `pages/DataFlywheel/` |
| Flutter 特性 | snake_case | `features/home/` |
| Spec 章节 | 数字_中文 | `01_预设计/` |
