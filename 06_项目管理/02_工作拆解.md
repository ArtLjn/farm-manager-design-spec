# 02 — 工作拆解

> 状态：草稿 | 维护：BlockShip | 关联：[01_里程碑与路线图.md](./01_里程碑与路线图.md)、[../../docs/plans/current-sprint.md](../../docs/plans/current-sprint.md)

## 1. 拆解原则

- **目标 → 关键结果 → 任务**：每条任务必须能回溯到一个 KR。
- **任务粒度**：1 条任务 ≤ 2 天；> 2 天必须再拆。
- **DoD（Definition of Done）**：每条任务必须写清"完成定义"。
- **验收维度**：代码 / 测试 / 文档 / 监控 / 回滚方案 五选多。

## 2. 拆解层级

```
Phase (季度级)
   └── Epic (月级，对应一个 Phase 子目标)
        └── Story (周级，对应用户可感知价值)
             └── Task (天级，单人完成)
                  └── Subtask (小时级)
```

## 3. Epic / Story / Task 示例

以 Phase 4 "Reflector 完整化" 为例：

### Epic 4.1 Reflector 完整化

| Story | 描述 | DoD |
| --- | --- | --- |
| 4.1.1 幻觉检测 | 检测金额 / 日期 / 实体与工具结果不一致 | Reflector 覆盖率 100%，幻觉率 < 5% |
| 4.1.2 工具一致性 | 校验 Skill 返回 vs LLM 复述 | 单元测试 + 仿真集 |
| 4.1.3 Prompt 泄露 | 检测响应中含 system prompt 片段 | 对抗集拦截率 100% |
| 4.1.4 异常回复告警 | Reflector 触发告警链路 | Grafana 告警规则 + 演练 |

### Story 4.1.1 拆解为 Task

| Task | 工时 | DoD |
| --- | --- | --- |
| 4.1.1.1 设计 Reflector 输入 schema | 0.5d | Pydantic 模型 + 测试 |
| 4.1.1.2 实现金额一致性 check | 0.5d | 函数 + 3 个单测 |
| 4.1.1.3 实现日期一致性 check | 0.5d | 函数 + 3 个单测 |
| 4.1.1.4 实现实体（人名 / 地块）check | 1d | 函数 + 3 个单测 |
| 4.1.1.5 集成到 Runtime graph | 0.5d | 集成测试 + trace |
| 4.1.1.6 文档 + Skill 清单更新 | 0.5d | 本 Spec + docs/ 同步 |

## 4. 任务字段

每条 Task 至少包含：

```yaml
id: 4.1.1.2
title: 实现金额一致性 check
owner: <github-username>
estimate: 0.5d  # 工时估算
phase: 4
epic: 4.1
story: 4.1.1
depends_on: [4.1.1.1]
status: todo  # todo | doing | review | done | blocked
doD:
  - code: 实现 check_amount_consistency() 函数
  - test: 3 个单测覆盖正/负/边界
  - docs: 更新 [01_正式设计/01_Agent平台架构] Reflector 章节
  - monitor: 添加 reflector_amount_mismatch 指标
  - rollback: 通过 feature flag 关闭 Reflector 即可回滚
priority: P0
sprint: 2026-W26
```

## 5. 优先级

| 等级 | 含义 |
| --- | --- |
| P0 | 阻塞发布 / 安全 / 数据丢失 |
| P1 | 核心 KR 必须项 |
| P2 | 重要但可延后一个 sprint |
| P3 | 锦上添花 |

## 6. 工时估算参考

| 工作类型 | 经验值 |
| --- | --- |
| 新增 Skill（含 skill.md + tests） | 1-1.5d |
| 新增 API 端点（含 schema + service + tests） | 1d |
| LangGraph 节点修改 | 0.5-1d |
| Prompt 重写 + 评测 | 1-2d |
| 数据库迁移 | 0.5-1d |
| Bug 修复（小） | 0.5d |
| Bug 修复（大） | 1-2d |
| 文档同步 | 0.5d |

## 7. Sprint 节奏

- **周期**：2 周。
- **节奏**：
  - 第 1 天：Sprint Planning，认领任务。
  - 每天：Daily Standup（10 分钟，异步可）。
  - 第 10 天：Sprint Review + Demo。
  - 第 11 天：Sprint Retro + 下个 Planning。

- **容量**：
  - 每人每 sprint 净开发时间 ≈ 6-8 天（扣除会议 / oncall / 杂事）。
  - Buffer 20% 应对临时插入（线上 bug / 紧急支持）。

## 8. 任务管理工具

| 用途 | 工具 |

## 9. Definition of Done（DoD）模板

每条 Story 进入 `done` 状态前，必须满足：

- [ ] 代码通过 `ruff check . && ruff format .`
- [ ] 代码通过 `pytest -v`
- [ ] 代码通过 `bash scripts/check-layer-deps.sh`
- [ ] 单文件 ≤ 500 行，单方法 ≤ 50 行
- [ ] 新增功能有对应测试（覆盖率达标）
- [ ] 文档已同步（按 [04_相关规范/06_防污染与文档同步规范](../04_相关规范/06_防污染与文档同步规范.md) 第 2.1 节）
- [ ] 没有 console.log / print 调试残留
- [ ] 没有未删除的 TODO
- [ ] 没有 .env / 密钥提交
- [ ] Commit message 符合 Conventional Commits
- [ ] PR 至少一个 reviewer approve
- [ ] 部署后监控指标无异常（24h）

## 10. 工作拆解反模式

- 任务粒度过粗（"完善 Agent" → 无法 estimate）。
- 缺 DoD（"做完"定义不清）。
- 跨 Story 共享 Task（造成依赖混乱）。
- 不留 Buffer（满负载导致延期）。
- 只列开发不列测试 / 文档 / 演练。
