# 08 — 业务模块化

> 状态：草稿 | 维护：BlockShip | 关联：[01_Agent平台架构](./01_Agent平台架构.md)、[../../docs/architecture/boundaries.md](../../docs/architecture/boundaries.md)

## 1. 为什么需要业务模块化

当前业务逻辑散落在 `app/api/` + `app/services/` + `app/models/`，没有清晰边界。问题：
- 跨域耦合（crop 调用 cycle 的内部方法）
- 责任不清（ledger 算利润的逻辑在哪）
- 难以演进（改 weather 要看懂全后端）

目标：**每个业务域一个 `modules/<domain>/` 包，对外只暴露 router + service + port**。

## 3. 模块内部结构（强制）

```
modules/<domain>/
├── __init__.py
├── router.py              # FastAPI router，仅编排
├── service.py             # 业务逻辑
├── dependencies.py        # FastAPI Depends（如 current_user、current_farm）
├── ports.py               # 对外接口定义（Protocol）
├── schemas.py             # Pydantic schemas（或从 app/schemas/ 迁入）
├── errors.py              # 模块错误码
├── repositories/          # （可选）数据库访问层
└── tests/
    ├── test_service.py
    ├── test_router.py
    └── conftest.py
```

**最小落地要求**：`router.py` / `service.py` / `dependencies.py` / `ports.py` 至少一类真实职责落地才创建目录，禁止空壳。

## 4. 依赖方向

```
api/ → modules/<domain>/router.py → modules/<domain>/service.py
                                          ↓
                                    modules/<domain>/ports.py（对外协议）
                                          ↓
                            其他 modules 通过 ports 调用（禁止 import service 内部）
                                          ↓
                                    core / infra / models
```

**跨模块调用规则**：
- 必须通过 `ports.py` 定义的 Protocol
- 禁止直接 import 另一模块的 `service.py`
- 禁止直接读另一模块的 model 表

## 5. 已迁移模块

### 5.1 Auth 模块（`modules/auth/`）

| 文件 | 职责 |
| --- | --- |
| `router.py` | 注册、登录、刷新 token、修改密码 |
| `service.py` | 密码 hash、token 签发/校验 |
| `dependencies.py` | `current_user`、`require_permission` |
| `permissions.py` | 权限矩阵 |
| `errors.py` | 错误码：401、403、404 |
| `schemas.py` | LoginRequest、TokenResponse 等 |

**禁止**：
- 把 token 解析逻辑放到其他模块
- 在 Auth 中操作农场业务规则（创建默认农场走 Farm 模块）

### 5.2 Farm 模块（`modules/farm/`）

| 文件 | 职责 |
| --- | --- |
| `router.py` | 农场 CRUD、设置 |
| `service.py` | 农场创建、默认农场逻辑 |
| `dependencies.py` | `current_farm` |
| `ports.py` | 其他模块获取农场信息的端口 |

**禁止**：
- 把 crop/cycle 业务规则放到 Farm
- 直接绕过 Farm 模块操作 farm 表

## 7. 迁移原则

1. **不停服**：模块迁移分多 PR，每次只搬一个域
2. **先 port 后 router**：先定义 `ports.py`，让其他模块先迁到 port，再迁 router 和 service
3. **保留兼容入口**：旧 `services/cost_service.py` 可保留 30 天作为 re-export
4. **测试先行**：迁移前必须有覆盖该域的回归测试
5. **CI 验证**：`scripts/check-layer-deps.sh` 验证迁移后的依赖方向

## 8. Skill 与业务模块的关系

Skill 是 Agent 调用业务模块的「翻译层」：

```
LLM 决定调用 create_cost_record
  ↓
Skill create_cost_record.execute()
  ↓
调用 modules/ledger/ports.py: LedgerPort.record_cost()
  ↓
modules/ledger/service.py 真正写库
```

**Skill 不直接访问数据库表，必须通过模块的 port**。这样 Skill 层只关心：
- 参数推断（从自然语言到 schema）
- 错误处理（DB 失败如何回复用户）
- 写操作确认（Pending Action）

业务规则的正确性由模块 service 保证。
