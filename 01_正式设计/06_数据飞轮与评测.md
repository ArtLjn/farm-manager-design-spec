# 06 — 数据飞轮与评测

## 1. 为什么需要数据飞轮

Farm Manager 不是一次性交付的产品，而是持续进化的 AI 系统。每一轮真实对话都比评测集宝贵 10 倍且免费。**数据飞轮是从真实会话、trace、event log 和仿真失败中提取样本，经过规则候选 → LLM 预标注 → 人工确认 → 回归/评测/训练数据的闭环**。

这是中小团队对抗大厂的唯一路径。

## 2. 完成态（北极星）

工业级数据飞轮不是赞踩按钮，也不是日志列表。它是一条持续转动的数据链路：

```
真实会话 / Playground / Simulation 失败
  → MySQL 热索引 + JSONL 原始事件
  → 规则初筛
  → LLM 自动预标注
  → 人工确认和根因标注
  → Bad Case / Tool Selection / Pending Safety / SFT 数据集
  → 生成 regression / evaluation case
  → Simulation 回归运行
  → Evaluation 汇总趋势
  → 修 prompt / router / skill / pending plan
  → 新会话继续回流
```

四个硬要求：
- **快速发现**：自动捞出工具格式异常、幻觉执行、pending 漏拦截等坏例
- **证据完整**：每个样本能回溯 session_id、turn_id、request_id、trace、tool I/O、pending lifecycle、token、latency、prompt/model 版本
- **标注可信**：AI 预标注可用，最终真值必须来自人工确认或确定性规则
- **闭环可消费**：标注样本能进入 regression case、evaluation replay、router 训练、pending safety、SFT JSONL

## 3. 模块边界

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| **Playground** | 手工发起对话、复制 debug JSON、复现用户输入 | 批量样本治理 |
| **TraceMonitor** | 定位单次请求链路、节点 I/O 和耗时 | 标注队列和数据集版本管理 |
| **DataFlywheel** | 样本队列、根因标注、AI 预标注采纳、导出、生成 case 草稿 | 实际执行回归测试 |
| **Simulation** | 主动运行 regression case，验证当前 Agent 行为 | 真实会话样本筛选和人工标注 |
| **Evaluation** | 汇总通过率、工具选择准确率、pending 安全趋势、版本对比 | 单条 trace 调试 |
| **Prompt/Router/Skill** | 消费飞轮产出的修复信号并改进行为 | 直接写入标注真值 |

## 4. 数据源

| 数据源 | 示例 | 用途 |
| --- | --- | --- |
| **显式用户反馈** | 赞、踩、Retry、主动纠错 | 判断回复质量和用户满意度 |
| **隐式用户反馈** | 复制回答、继续追问、中途退出、重复问 | 辅助判断回答有用性 |
| **Agent 内部链路** | selected tools、actual tool calls、pending plan、tool error、token、latency | 定位技术根因和回归 case |
| **仿真与评测回流** | simulation failed、evaluation replay failed | 防回归 |

显式反馈稀疏，**不能作为唯一来源**。必须依赖 Agent 内部链路和仿真回流补齐覆盖率。

## 5. 自动标注三层

按可信度递增：

| 层 | 来源 | 可作为真值 |
| --- | --- | --- |
| 1. 规则候选 | `rule`（确定性事件 + 文本规则） | 仅高置信规则可入候选，真值仍建议人工确认 |
| 2. LLM 预标注 | `llm_judge`（读样本证据 + debug JSON，输出 quality + root_cause + severity + confidence + label + reason） | ❌ 不可直接作为真值 |
| 3. 人工最终标注 | `human`（采纳/修改/驳回 AI 建议） | ✅ 真值 |

**关键约束**：同一个模型不能既生产线上回复又把自己的评分作为最终真值。同模型 judge 只能作为预标注和排序信号。

## 6. 标签体系（固定枚举）

避免标签发散：

| 标签 | 含义 | 维度 |
| --- | --- | --- |
| `good_reply` | 好回复 | 输出 |
| `bad_reply` | 坏回复 | 输出 |
| `wrong_tool_selection` | 工具选错 | 决策 |
| `tool_parameter_mismatch` | 工具参数、实体、数量或作用域与用户意图/上下文不一致 | 决策 |
| `pending_missed` | pending 漏拦截 | 安全 |
| `hallucinated_execution` | 幻觉执行（回复说做了但 tool_call 没成功） | 一致性 |
| `tool_error_ignored` | 工具失败后回复仍称成功 | 一致性 |
| `off_topic` | 答非所问 | 输出 |
| `sensitive_info_leak` | 参数、提示词、内部信息泄露 | 安全 |
| `missing_wage` | 安排农事但工资缺失 | 业务规则 |
| `disabled_worker_used` | 禁用工人被使用 | 业务规则 |
| `unclear_intent` | 用户输入不足 | 输入 |
| `not_actionable` | 暂不处理 | 元 |
| `needs_regression` | 需沉淀为回归用例 | 元 |

标注时区分：
- 用户输入是否可处理（如 `unclear_intent`）
- Agent 输出是否合格（如 `bad_reply`、`hallucinated_execution`）

## 7. 存储与资源约束

小服务器部署轻量架构：

| 存储 | 用途 | 查询模式 |
| --- | --- | --- |
| **MySQL** | 热索引、turn 元数据、标签、case draft、dataset 版本、评测摘要 | 列表页 |
| **JSONL** | 原始事件、tool payload、trace segment、debug evidence | 详情页（按 event_file + seq range 局部读） |

**禁止**：
- 列表页扫全量 JSONL
- 引入 Kafka / ClickHouse / Mongo / 数据湖
- 同步导出大体积数据（必须异步任务）

数据保留：
- 热索引保留 90 天，列表检索
- 原始 JSONL 按日期/farm/session 分区，保留 1 年
- 导出训练数据前必须脱敏

### 7.1 Agent Events JSONL 字段字典与准确性保障

示例文件：
`backend/data/agent-events/dt=2026-06-23/farm_id=2/session_id=playground-1782198580963-g8l73s/events.jsonl`

该文件是一次 session 的 append-only 事件流。列表页只读取 MySQL 热索引；详情页按 `event_file + event_seq_range` 局部读取 JSONL。

顶层字段：

| 字段 | 含义 | 来源 | 准确性要求 |
| --- | --- | --- | --- |
| `event_id` | 单条事件唯一 ID | 写入事件时生成 UUID | 不参与业务关联，只用于去重和审计 |
| `event_type` | 事件类型，如 `message.user`、`message.assistant` | 写入方显式传入 | 必须与 payload 结构匹配 |
| `schema_version` | 事件 schema 版本 | `AgentEventWriter` 固定写入 | schema 变更必须递增版本 |
| `created_at` | 事件写入时间，UTC ISO8601 | 事件写入时钟 | 用于排序参考，真实轮次顺序以 `seq` 为准 |
| `farm_id` | 农场 ID | 当前请求上下文 / conversation | 必须与 session 所属 farm 一致 |
| `user_id` | 用户 ID，助手消息可为空 | 当前登录用户 / conversation | 用户消息必须有值；助手消息允许 `null` |
| `session_id` | 会话 ID | 当前请求或 conversation | 文件路径中的 session 分区必须一致 |
| `turn_id` | 聚合后的对话轮次 ID | `agent_turns.id` | 必须能关联到 `agent_turns` |
| `request_id` | 单次请求链路 ID | chat 请求生成；回填为 `bf...` | 用于跳 TraceMonitor；回填 ID 不能伪装成真实 trace |
| `seq` | session 文件内递增序号 | 写入文件时按文件递增 | 同一 events.jsonl 内必须单调递增且不重复 |
| `payload` | 事件负载 | 按 `event_type` 写入 | 不同事件类型有不同结构 |

`message.user.payload`：

| 字段 | 含义 | 来源 | 准确性要求 |
| --- | --- | --- | --- |
| `content` | 用户原始输入 | 请求入参 / conversation message | 必须与 `conversation_messages.content` 一致 |
| `message_id` | 用户消息表 ID | `conversation_messages.id` | 必须能回查到对应 user message |
| `backfilled` | 是否由历史会话补齐 | 仅同步补齐服务写入 | 实时事件没有该字段；回填事件不能用于判断实时 trace 完整性 |

`message.assistant.payload`：

| 字段 | 含义 | 来源 | 准确性要求 |
| --- | --- | --- | --- |
| `content` | 助手最终回复 | Agent 返回结果 / conversation message | 必须与 `conversation_messages.content` 一致 |
| `message_id` | 助手消息表 ID | `conversation_messages.id` | 必须能回查到对应 assistant message |
| `skills` | 本轮响应关联的 skill 名称列表 | chat use case / pending response | 只能表示响应声明的 skills，不等价于真实 tool call 全量 |
| `pending_action` | 单步写操作确认动作 | pending action response | 非空时必须包含 `action_id`、`skill_name`、`params` |
| `pending_plan` | 多步写操作确认计划 | pending plan response | 非空时必须能关联 pending plan 存储 |
| `backfilled` | 是否由历史会话补齐 | 仅同步补齐服务写入 | 回填助手事件通常没有实时 `skills/pending` 细节 |

`pending_action` 子字段：

| 字段 | 含义 | 准确性要求 |
| --- | --- | --- |
| `action_id` | pending 动作 ID | 必须能在 pending action 生命周期中定位 |
| `skill_name` | 待确认执行的写 skill | 必须是可执行 skill 名称 |
| `params` | 待执行参数 | 必须是最终等待用户确认的参数，不是模型草稿 |
| `context.original_input` | 触发 pending 的用户原话 | 必须与当前 turn 用户输入一致或可解释 |
| `context.extracted_params` | 参数抽取结果 | 必须与 `params` 对齐 |
| `context.notes` | 给用户展示的理解说明 | 仅作解释，不作为执行真值 |

为了保证字段能拿到真实结果，必须满足以下约束：

1. **同源写入**：消息入库、`agent_turns` 聚合和 JSONL 写入必须在同一请求链路中完成；不能由前端拼装事件。
2. **双写校验**：每个 turn 保存 `event_file`、`event_seq_start`、`event_seq_end`、`event_write_status`；详情页按该范围读取事件，读不到则标记“事件缺失”。
3. **DB 交叉校验**：`payload.message_id` 必须能回查 `conversation_messages`；`turn_id/request_id/session_id/farm_id` 必须与 `agent_turns` 一致。
4. **真实工具结果另存证据**：`skills` 只表示响应关联 skill。真实 `selected_tools`、`actual_tools`、tool input/output、router JSON、db diff 必须来自 trace/tool events 或专门事件，不能从回复文本反推。
5. **回填显式标记**：历史补齐事件必须写 `payload.backfilled=true` 和 message meta 的 `event_backfilled=true`，避免把回填数据误判为实时完整链路。
6. **顺序以 seq 为准**：同一 session 文件内按 `seq` 判断顺序；`created_at` 只做时间展示和异常排查。
7. **缺失不可静默**：缺少 event、trace、pending lifecycle、tool result、db diff 时，飞轮 UI 必须展示缺失项，不能用空值当作“无工具/无事件”。

## 8. 分阶段演进

### P0 — 证据完整化（当前）

目标：每轮会话能被复盘。

交付：
- `agent_turns`、`conversation_messages`、trace、JSONL event log 对齐
- debug export v2（导出本轮 skill/router/tool/pending/messages 上下文）
- pending plan 持久化（重启后仍能确认/取消/过期）
- DataFlywheel 详情页展示缺失证据段

验收：
- 一条多意图对话能从 DataFlywheel 跳 TraceMonitor，复制完整 debug JSON
- 写操作是否经过 pending、是否执行成功、是否影响数据库可被复查

### P1 — 人工标注闭环

目标：坏例能进入回归草稿。

交付：
- DataFlywheel 样本队列、详情、固定标签、备注、session 级标注
- 规则候选队列：工具格式异常、工具失败、幻觉执行、pending 漏拦截、写操作误触发
- 上下文一致性候选：工具参数实体、数量、作用域与前文候选集合不一致时自动入队
- 当前样本 JSONL 导出
- `case draft` 生成（含 expected skills、expected pending、confirmation flow、reply assertions、expected db diff）
- 人工确认后加入 Simulation 或 Evaluation replay

### P2 — AI 自动预标注

目标：降低人工标注成本，不牺牲真值质量。

交付：
- LLM judge prompt + 版本管理
- 预标注 API（输入样本证据 + 前后文窗口 + 工具参数 + pending 动作 + 可选数据库差异，输出 suggested labels / root cause / severity / confidence / reason / recommended fix）
- 前端展示「AI 预判」，支持采纳/修改/驳回
- 标注来源记录（`rule` / `llm_judge` / `human`），保留 judge model + prompt version
- 低置信度入人工优先队列，高置信规则入快速确认队列

### P3 — Dataset 与仿真评测闭环

目标：修复效果可量化、可防回归。

交付：
- 数据集版本管理（dataset_name / version / split / source_sample_id / label_source）
- DB-backed simulation cases（避免写部署包内 JSON）
- Simulation 失败结果自动回流到 DataFlywheel
- Evaluation 汇总版本趋势（通过率、工具选择准确率、pending 漏拦截率、幻觉执行率、token/latency）
- Prompt/router/skill 改动必须跑核心 regression suite

### P4 — 训练与调优出口

目标：高质量标注数据 → 模型 / prompt / router / skill 改进输入。

交付：
- SFT JSONL 导出（user input + 必要上下文 + tool result + 人工修正回复）
- Router 训练/评测样本（input + expected tools + rejected tools + reason）
- Pending safety 数据集（写操作 + 确认话术 + 应拦截/应执行 + 期望 db diff）
- Prompt A/B 与 pairwise 回复对比
- 修复建议归档（每个 bad case 关联修复 PR / prompt version / skill version）

## 9. 当前飞轮功能需求基准

> 本节只保留产品口径。页面细节见 §10，发现层见 §11，审核对象见 §12。后续实现计划必须先对齐本节，避免继续出现“既按 session 又按 turn”的双主线。

### 9.1 设计原则

1. **Session-first**：标注员默认处理的是一个 session 内的问题链，不是孤立 turn。
2. **Turn-as-evidence**：turn 是证据单元，用于展示上下文、工具、pending、trace、结果，不作为主工作台入口。
3. **IssueChain-as-task**：真正的审核任务是 `ReviewIssueChain`，它由 trigger turn、context turns、result turns 组成。
4. **Evidence-before-label**：没有证据说明“为什么坏”，就不能只展示风险分数让人猜。
5. **Expected-before-repair**：没有 expected behavior，不能生成 regression draft，也不能导出 repair pack。
6. **Judge-is-advice**：AI 预判只能给建议和排序，不能直接成为最终真值。

核心约束：默认只进每日质检；全部 turn 只能在高级搜索里查，不作为主流程。

### 9.2 页面边界

| 页面 | 主对象 | 核心作用 | 不负责 |
| --- | --- | --- | --- |
| **每日质检** | `ReviewIssueChain` / `session_id` | 标注员每天处理风险会话、查看上下文、做人工判断、沉淀 expected behavior | 全量日志检索、批量导出训练集 |
| **高级搜索** | `session` / `turn` / `request_id` | 查原始样本、定位 request、按条件抽检 turn、创建/补充问题链候选 | 默认审核入口；最终人工标注 |
| **修复包** | `RepairPack` | 管理已确认问题的修复任务、证据、验收标准和状态 | 判断样本是否坏 |
| **数据集 / 评测** | `DatasetVersion` / `EvalRun` | 管理 regression case、dataset 版本、评测趋势、失败回流 | 人工逐条标注 |
| **TraceMonitor** | `request_id` / trace segment | 深挖单次请求链路、工具 I/O、耗时和错误 | 承担标注队列 |

旧入口收敛：

| 旧入口/概念 | 新定位 |
| --- | --- |
| `全部 turn` | 降级到 **高级搜索**；仅用于定位、抽检、debug，不出现在默认主流程 |
| `问题候选` | 作为 **每日质检 Inbox 的筛选来源**，不独立成为主工作台 |
| `Session 复盘` | 成为每日质检中间的 **Session Timeline** |
| `Turn 审核` | 成为 Timeline 中单个 turn 的展开详情，或高级搜索里的单 turn 详情 |
| `样本详情 / 标注` | 重构为右侧 **问题链审核面板** |

### 9.3 必须能力

- 每日质检：左侧风险 session / issue chain，中心 session timeline，右侧审核面板。
- 高级搜索：保留 session、request、turn、标签、工具、风险分检索，但不作为默认首页。
- 修复包：只能从已确认且证据足够的问题链生成，必须带 expected behavior。
- 数据集/评测：从 accepted issue chain 生成 regression draft，保留 `chain_id` 和 `related_turns`。
- 审计治理：记录标签来源、reviewer、judge prompt/model version；训练集导出前必须脱敏。

### 9.4 问题发现能力

| 来源 | 发现什么 | 必须输出 |
| --- | --- | --- |
| Rule Engine | 工具失败、幻觉执行、pending 漏拦截、敏感信息、明确业务规则 | rule_id、suggested label、severity、reason、evidence |
| Context Analyzer | 批量意图被收窄、指代丢失、参数作用域错配、pending 参数与上下文不一致 | candidate_type、context facts、mismatch detail、source turns |
| LLM Judge | 答非所问、意图理解错、综合证据后的根因建议 | labels、root_cause、confidence、reason、recommended_fix、prompt_version |

`repair-manual_triage-2ced95bdeea1` 这类问题必须能被 Context Analyzer 或 Judge 候选覆盖：

- 前文有 7 个未付工人；
- 用户说“所有员工工资结了”；
- pending / 执行参数只保留单个工人；
- 候选标签应至少包含 `tool_parameter_mismatch` 和 `needs_regression`；
- 诊断必须说明“批量作用域被收窄”，不能只显示 `Rule Risk: 0.95`。

### 9.5 MVP 优先级

| 优先级 | 功能 | 验收标准 |
| --- | --- | --- |
| P0 | 默认进入每日质检，不再默认展示全部 turn | 标注员打开页面先看到风险 session / issue chain |
| P0 | Inbox 卡片展示 why + evidence status | 不需要点详情就知道为什么候选、缺什么证据 |
| P0 | Session Timeline 高亮 context / trigger / result | 能看懂一个坏例的上下文链路 |
| P0 | 右侧审核面板 | 能完成 accepted / rejected / needs_evidence / not_actionable |
| P0 | expected behavior 必填约束 | accepted 后能沉淀正确行为 |
| P1 | issue chain → regression draft | 回归草稿保留 chain_id 和 related turns |
| P1 | issue chain → repair pack | 修复包能追溯原始证据和 expected behavior |
| P1 | 高级搜索承接全部 turn | 原始 turn 查询能力保留，但不干扰主流程 |
| P2 | Context Analyzer v1 | 覆盖批量意图收窄、指代丢失、pending 参数错配 |
| P2 | LLM Judge evidence pack | AI 预判基于 session 上下文，而不是单 turn |
| P3 | dataset / evaluation 闭环 | 修复后能评测、防回归、失败回流 |

### 9.6 明确不做

- 不把 `全部 turn` 作为默认首页。
- 不让 AI Judge 直接写最终真值。
- 不只展示风险分数而不展示原因和证据。
- 不允许缺 expected behavior 的样本进入回归或修复包。
- 不引入 Kafka、ClickHouse、向量库等新基础设施。
- 不做无边界的全量 JSONL 列表扫描。

## 10. 页面详细功能设计

> 本节只描述页面功能，不描述后端实现。每个页面必须有明确主对象、入口、核心区域、动作和禁用条件。

### 10.1 每日质检页

页面目标：让标注员按风险处理 `ReviewIssueChain`，在一个页面内完成“找问题、看上下文、做判断、写 expected behavior、进入闭环”。

主对象：`ReviewIssueChain`。页面可以显示 session，但审核任务必须落到具体 issue chain。

```text
+--------------------------------------------------------------------------------------------------+
| 每日质检                                                                                         |
| 筛选：日期 farm severity 状态 信号 证据状态 标签  [刷新] [同步事件]                              |
+----------------------------+---------------------------------------------+-----------------------+
| 左：风险队列               | 中：Session Timeline                         | 右：审核面板          |
+----------------------------+---------------------------------------------+-----------------------+
| 队列分组                   | Session 摘要                                 | 当前 IssueChain       |
| - P0/高风险                | - session_id / farm / user / 时间            | - 标题 / 类型 / P0    |
| - 证据不足                 | - 当前链数量 / 已处理数量                    | - 状态 / 主导信号     |
| - 待人工确认               |                                             |                       |
|                            | Timeline turn 卡片                           | 证据 checklist        |
| Session/Chain 卡片         | - context / trigger / result / unrelated     | - present             |
| - session_id               | - user input                                 | - missing             |
| - 用户输入摘要             | - assistant reply                            | - needs_human         |
| - 问题链标题               | - selected_tools / actual_tools              |                       |
| - 风险分 / severity        | - pending_action / pending_plan              | AI 预判               |
| - 证据状态                 | - event / trace / request_id                 |                       |
| - 下一步动作               |                                             | 人工判断              |
|                            | Turn 展开层                                  | - accepted/rejected   |
|                            | - messages                                   | - needs_evidence      |
|                            | - router JSON                                | - not_actionable      |
|                            | - tool input/output                          |                       |
|                            | - pending lifecycle                          | Expected behavior     |
|                            | - db diff                                    |                       |
|                            |                                             | 闭环出口              |
|                            |                                             | [保存] [下一条]       |
|                            |                                             | [生成回归] [修复包]  |
+----------------------------+---------------------------------------------+-----------------------+
```

左侧风险队列：

| 功能 | 具体要求 |
| --- | --- |
| 队列来源 | rule、context analyzer、LLM judge、evaluation failed 回流 |
| 默认排序 | `severity DESC, risk_score DESC, created_at DESC` |
| 筛选 | 日期、farm、severity、状态、主导信号、证据状态、标签、reviewer |
| 卡片字段 | session_id、用户输入摘要、问题链标题、trigger turn、风险分、severity、状态、证据状态、候选原因 |
| 卡片动作 | 打开链、标记误报、标记暂不处理、补证据、跳高级搜索 |
| 空状态 | 告诉标注员当前筛选无待处理链，并给出“放宽筛选/同步事件”动作 |

中间 Session Timeline：

| 功能 | 具体要求 |
| --- | --- |
| 时间线范围 | 默认展示当前 session 全部 turn；长 session 折叠无关 turn |
| turn 角色 | `context`、`trigger`、`result`、`unrelated` |
| turn 摘要 | user input、assistant reply 摘要、tools、pending、request_id、token、latency |
| 展开详情 | messages、router JSON、tool input/output、pending lifecycle、event JSON、trace 链接、db diff |
| 人工调整 | 可把 turn 加入/移出 context/result；不能让同一 turn 同时属于 context 和 result |
| 缺失证据 | event、trace、tool result、db diff 缺失时必须显示缺失标签 |

右侧审核面板：

| 区块 | 字段/动作 |
| --- | --- |
| 诊断摘要 | title、candidate_type、severity、dominant_signal、suggested_labels、reason |
| 证据 checklist | evidence key、状态、来源 turn、缺失原因、补证据动作 |
| AI 预判 | labels、confidence、root_cause、reason、recommended_fix、model、prompt_version |
| 人工判断 | review_decision、final_labels、root_cause、fix_target、expected_behavior、reviewer_comment |
| 闭环出口 | 保存并下一条、生成 regression draft、导出 repair pack、跳 TraceMonitor |

人工判断字段必须分层，不能把“处理状态”和“问题标签”混在一起：

| 字段 | 枚举/来源 | 含义 |
| --- | --- | --- |
| `review_decision` | `accepted` / `rejected` / `needs_evidence` / `not_actionable` | 人工对这个候选链的处理结论 |
| `final_labels` | 使用 § 6 固定标签，可多选 | 人工确认后的质量/根因标签 |
| `root_cause` | 人工文本，必填条件见下表 | 说明为什么错，不能只写标签 |
| `fix_target` | `prompt` / `router` / `tool` / `pending` / `data` / `policy` / `unknown` | 预计修复归属，用于修复包分流 |
| `expected_behavior` | 人工文本 | 正确行为，生成 regression / repair pack 的前置条件 |
| `reviewer_comment` | 人工文本 | 误报原因、补证据说明或其他备注 |

`final_labels` 可选范围：

| 标签 | 何时使用 |
| --- | --- |
| `good_reply` | 人工确认回复没问题，通常配合 `rejected` 表示误报 |
| `bad_reply` | 回复整体质量差，但暂不能归到更具体错误 |
| `wrong_tool_selection` | 应调用/不应调用/调用了错误工具 |
| `tool_parameter_mismatch` | 工具参数、实体、数量、作用域和用户意图或上下文不一致 |
| `pending_missed` | 写操作应该确认但没有 pending |
| `hallucinated_execution` | 回复声称已执行，但没有真实成功执行证据 |
| `tool_error_ignored` | 工具失败后仍回复成功 |
| `off_topic` | 答非所问 |
| `sensitive_info_leak` | 泄露内部参数、提示词或敏感信息 |
| `missing_wage` | 安排农事但工资缺失 |
| `disabled_worker_used` | 使用了禁用工人 |
| `unclear_intent` | 用户输入不足，正确行为应追问 |
| `not_actionable` | 暂不处理或无法形成修复/回归资产 |
| `needs_regression` | 需要沉淀为回归用例 |

禁用条件：

| 动作 | 禁用条件 |
| --- | --- |
| 生成 regression draft | 未 accepted；缺 expected behavior |
| 导出 repair pack | `needs_evidence`；缺 expected behavior；没有 trigger turn |
| accepted 保存 | 缺 final labels、root cause 或 expected behavior |
| rejected 保存 | 缺误报原因 |
| needs_evidence 保存 | 没有选择缺失证据类型或补证据说明 |

验收标准：

- 标注员不需要打开 `全部 turn` 表格，也能完成一个坏例审核。
- `repair-manual_triage-2ced95bdeea1` 这类跨轮参数错配能在 timeline 中看到 context、trigger、result。
- 右侧能明确显示为什么候选、证据缺什么、正确行为是什么。

### 10.2 高级搜索页

页面目标：查证、抽检、定位原始样本。它不是默认标注入口，也不写最终人工标注。

主对象：`session`、`request_id`、`turn`。

```text
+--------------------------------------------------------------------------------+
| 高级搜索                                                                       |
| 搜索框：session_id / request_id / turn_id / 用户输入关键词                     |
| 筛选：日期 farm 标签 工具 状态 风险分 是否已入链                               |
+--------------------------------------+-----------------------------------------+
| 左：结果列表                         | 右：原始详情                            |
+--------------------------------------+-----------------------------------------+
| 结果行                               | 基础信息                                |
| - session_id                         | - farm / user / session / turn / request |
| - turn_id / request_id               | - event_file / seq range                |
| - 用户输入摘要                       |                                         |
| - 标签 / 风险分 / 状态               | 原始消息                                |
| - 是否已关联 IssueChain              | - user content                          |
|                                      | - assistant content                     |
| 批量动作                             |                                         |
| - 批量 AI 分析                       | 证据详情                                |
| - 批量加入候选                       | - event JSON                            |
| - 导出 Evidence Pack                 | - trace link                            |
|                                      | - selected/actual tools                 |
|                                      | - pending lifecycle                     |
|                                      |                                         |
|                                      | 动作                                    |
|                                      | [打开 Session Timeline]                 |
|                                      | [创建候选链] [补充到已有候选链]        |
|                                      | [导出 Debug JSON]                       |
+--------------------------------------+-----------------------------------------+
```

页面边界：

| 能做 | 不能做 |
| --- | --- |
| 搜索 session / request / turn | 不能把 turn 直接标为 accepted/rejected |
| 查看原始 messages、event、trace、tool、pending 证据 | 不能写 final_labels |
| 从 turn 创建 `ReviewIssueChain` 候选 | 不能填写最终 root_cause |
| 把 turn 补充到已有候选链的 context/result | 不能填写 expected_behavior |
| 批量 AI 分析并生成候选建议 | 不能生成 regression draft 或 repair pack |
| 导出 Debug JSON / Evidence Pack | 不能替代每日质检审核面板 |

功能要求：

| 功能 | 具体要求 |
| --- | --- |
| 搜索 | 支持 session_id、request_id、turn_id、关键词 |
| 筛选 | 日期、farm、标签、工具、状态、风险分、是否已标注、是否已入链 |
| 结果字段 | session_id、turn_id、request_id、输入摘要、回复摘要、工具数量、token、latency、风险分、标签 |
| 详情 | 原始 messages、event JSON、trace、tool I/O、pending、db diff |
| 转入主流程 | 能从 turn 创建 IssueChain 候选，或把 turn 补充到已有候选 chain |
| 导出 | Debug JSON、Evidence Pack；大批量导出必须异步 |

禁用条件：

- 批量 AI 分析结果不能直接写 final label，只能进入 issue candidate。
- 高级搜索中不能保存 `accepted`、`rejected`、`final_labels`、`root_cause`、`expected_behavior`。
- 缺 event range 时不能导出完整 Evidence Pack，只能导出“缺证据包”。

验收标准：

- 能通过 request_id 快速定位一次请求。
- 能看到该 turn 是否已属于某个 IssueChain。
- 能从原始 turn 回到每日质检的 session timeline。
- 所有最终标注动作只能在每日质检右侧审核面板完成。

| 左：Repair Pack 列表                 | 右：修复包详情                          |
+--------------------------------------+-----------------------------------------+
| Pack 卡片                            | Manifest                                |
| - pack_id                            | - pack_id / source_chain_id             |
| - 标题 / severity                    | - fix_target                            |
| - fix_target                         | - status / owner                        |
| - 状态 / owner                       |                                         |
| - regression_ready                   | 复现证据                                |
|                                      | - session_id                            |
|                                      | - trigger/context/result turns          |
|                                      | - 当前错误行为                          |
|                                      | - expected behavior                     |
|                                      |                                         |
|                                      | 修复建议                                |
|                                      | - root cause                            |
|                                      | - recommended fix                       |
|                                      | - 验收标准                              |
|                                      |                                         |
|                                      | 验证                                    |
|                                      | - regression case                       |
|                                      | - eval result                           |
|                                      | [标记修复] [验证通过] [reopen]          |
+--------------------------------------+-----------------------------------------+
```

功能要求：

| 功能 | 具体要求 |
| --- | --- |
| 列表 | 按状态、fix_target、severity、owner、创建时间筛选 |
| Manifest | pack_id、source_chain_id、source_session、related_turns、fix_target、status |
| 证据 | 展示诊断摘要、root cause、expected behavior、原始证据链接 |
| 工程交接 | recommended fix、验收标准、建议测试命令、关联 PR/prompt version |
| 验证 | 关联 regression case 和 evaluation result |
| 状态 | draft、ready、in_progress、fixed、verified、reopened |

禁用条件：

- 没有 `source_chain_id` 不能创建修复包。
- `needs_evidence` 的链不能导出 ready 修复包。
- 没有 expected behavior 不能进入 `ready`。
- 验证失败必须 reopen，不能直接标记 verified。

验收标准：

- 工程侧只看修复包也能复现问题、理解正确行为、知道验收标准。
- 修复包能回跳到原始 IssueChain 和 Session Timeline。
- 修复后能关联 regression/eval 结果。

页面目标：管理从问题链沉淀出来的 regression case、dataset version 和评测结果。

主对象：`RegressionCase`、`DatasetVersion`、`EvalRun`。

+--------------------------------------+-----------------------------------------+

+--------------------------------------+-----------------------------------------+
| Regression Case 列表                | Case 详情                                |
| - case_id                           | - chain_id                               |
| - source_chain_id                   | - related_turns                          |
| - labels                            | - input / expected behavior              |
| - ready / last_result               | - expected tools/pending/db diff         |
|                                      |                                         |
| Dataset Version                     | Eval Report                              |
| - dataset_name                      | - pass_rate                              |
| - version / split                   | - tool_selection_accuracy                |
| - case_count                        | - pending_missed_rate                    |
|                                      | - parameter_mismatch_rate                |
| Eval Run                            | - failure_stage                          |

```

功能要求：

| 功能 | 具体要求 |
| --- | --- |
| Regression Case | 从 accepted IssueChain 生成，保留 chain_id、related_turns、expected behavior |
| Dataset Version | 管理 dataset_name、version、split、source、label_source、case_count |
| Eval Run | 记录模型/prompt/router/skill 版本、通过率、失败阶段、token、latency |

- 缺 `chain_id` / `related_turns` 的 case 不能进入正式 dataset。
- 失败样本未归因前不能关闭回流任务。

验收标准：

- 每个 case 都能追溯到 source_chain_id 和原 session。
- 每个失败结果都能回流到 DataFlywheel。
- 修复前后指标能在同一页面对比。

### 10.5 规则候选详细设计

规则候选不是独立页面，而是把确定性风险转成 `ReviewIssueChain` 候选的机制。它必须让标注员看到“为什么值得审、证据在哪里、缺什么证据”，不能只显示 `Rule Risk: 0.95`。

边界：

| 项 | 说明 |
| --- | --- |
| 负责 | 确定性命中、候选原因、证据摘要、建议标签、风险排序 |
| 不负责 | 最终人工真值、expected behavior、修复结论 |
| 展示位置 | 每日质检 Inbox、Session Timeline 标记、右侧证据 checklist |
| 输出对象 | `IssueCandidate`，再聚合成 `ReviewIssueChain` |

结构：

```text
events/messages/trace/pending/db diff
  -> Rule Engine / Context Rules
  -> IssueCandidate
  -> ReviewIssueChain
  -> 每日质检人工审核
```

候选字段：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选唯一 ID，建议由 `rule_id + session_id + trigger_turn_id` 派生 |
| `type` / `rule_id` | 候选类型和稳定规则 ID |
| `source` | `rule` / `context_rule` / `issue_detector` |
| `severity` / `risk_score` | 处理优先级和排序分 |
| `suggested_label` | 建议人工标签，必须属于 §6 标签集 |
| `reason` | 面向标注员的人类可读原因 |
| `evidence` | 结构化证据摘要，不只是一段文本 |
| `source_turn_ids` / `trigger_turn_id` | 证据 turn 和触发 turn |
| `evidence_status` | `complete` / `missing_event` / `missing_trace` / `missing_db_diff` / `needs_human` |
| `missing_evidence` | 缺失证据项 |
| `rule_version` | 规则版本或配置 hash |

首批规则：

| 规则 ID | 命中条件 | 建议标签 | 级别 | 关键证据 |
| --- | --- | --- | --- | --- |
| `tool_error_ignored` | 工具失败，但回复声称完成 | `tool_error_ignored` | P0 | failed tool、reply |
| `hallucinated_execution` | 无成功写工具或 pending，却声称已执行 | `hallucinated_execution` | P0 | selected/actual tools、reply |
| `pending_missed` | 写操作执行，但缺确认计划 | `pending_missed` | P0 | write tool、pending lifecycle |
| `unsafe_write_on_question` | 用户是查询/确认类问题，router 选择写操作 | `pending_missed` | P0 | input、selected tools |
| `wrong_tool_selection` | 用户需要实时/外部数据，但 router 未选工具 | `wrong_tool_selection` | P1 | input、selected tools |
| `sensitive_info_leak` | 回复泄露内部参数、密钥或 prompt | `sensitive_info_leak` | P0 | reply keyword |
| `disabled_worker_used` | 停用工人仍被安排或结算 | `disabled_worker_used` | P0 | tool result |
| `missing_wage` | 工人缺工资、已付、不计工资或欠款策略 | `missing_wage` | P0 | tool params/result |
| `bulk_intent_narrowed_to_single_entity` | 用户说全部/所有，但执行只保留单个实体 | `tool_parameter_mismatch` | P0 | context、pending、result |
| `referential_scope_lost` | “这些/他们/上面那些”被解析成部分实体 | `tool_parameter_mismatch` | P0 | context entities、params |
| `db_diff_smaller_than_requested_scope` | 实际 db diff 小于用户请求范围 | `tool_parameter_mismatch` | P0 | requested scope、db diff |

证据要求：

| 候选类型 | 必须证据 | 缺失状态 |
| --- | --- | --- |
| 工具失败 / 幻觉执行 | selected tools、actual tools、assistant reply、trace error | `missing_trace` / `missing_event` |
| pending 类 | selected write tools、pending lifecycle | `missing_event` |
| 业务规则类 | tool params/result 或事件结果 | `missing_trace` |
| 上下文作用域类 | context turns、pending params、result/db diff | `needs_human` / `missing_db_diff` |

去重与验收：

- 同一 `session_id + trigger_turn_id + type` 只保留一个候选。
- 多个规则命中同一 trigger turn 时合并到同一个 `ReviewIssueChain`，保留多条 evidence。
- 被人工驳回的候选再次命中时，不自动重新入队，除非规则版本或 evidence 变化。
- `repair-manual_triage-2ced95bdeea1` 这类批量作用域收窄问题必须生成上下文规则候选，并说明“批量作用域被收窄”。

### 10.6 AI 预判详细设计

AI 预判是 `ReviewIssueChain` 的辅助判断，不是线上业务回答的一部分。它应作为 DataFlywheel 的异步 Judge Agent 分支运行，不阻塞线上对话。

设计结论：

| 问题 | 结论 |
| --- | --- |
| 是否作为系统 Graph Agent 分支 | 是，但属于离线/异步 DataFlywheel Judge 分支 |
| 是否结合上下文 | 必须结合 session 上下文；禁止只看当前 turn 做最终预判 |
| 是否结合规则候选 | 必须结合；规则候选提供证据和注意力范围 |
| 是否能作为最终真值 | 不能，只能给建议、排序、root cause 草稿和 recommended fix |
| 预判对象 | 优先 `ReviewIssueChain`，不是孤立 turn |

结构：

```text
ReviewIssueChain + IssueCandidate
  -> Evidence Pack Builder
  -> Judge LLM Node
  -> Output Normalizer
  -> Workbench Projection
```

Evidence Pack 必填内容：

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `chain_id` / `session_id` | ReviewIssueChain | 标识对象并关联完整会话 |
| `trigger_turn` | messages/events | 判断问题发生点 |
| `context_turns` | 同 session 前文 | 判断指代、批量作用域、用户目标 |
| `result_turns` | 后续确认/执行结果 | 判断执行是否符合期望 |
| `issue_candidates` | 规则候选 | 提供候选类型、原因和证据 |
| `selected_tools` / `actual_tools` | router/trace | 判断工具选择和真实执行 |
| `pending_lifecycle` | pending events / DB | 判断写操作是否正确确认 |
| `tool_io` / `db_diff` | trace / audit | 判断参数、结果和影响范围 |
| `evidence_status` | 证据读取结果 | 告诉 Judge 哪些证据缺失 |

AI 预判输出：

| 字段 | 含义 |
| --- | --- |
| `judge_id` / `chain_id` | 本次预判和对象 ID |
| `labels` | 建议标签，必须来自 §6 |
| `root_cause` / `reason` | 建议根因和人类可读解释 |
| `severity` / `confidence` | 建议严重程度和置信度，仅用于排序提示 |
| `recommended_fix` | 修复建议草稿 |
| `evidence_refs` | 引用的 turn / event / trace |
| `missing_evidence` | 还缺什么证据 |
| `judge_model` / `prompt_version` | 可追溯运行信息 |

结合规则候选的行为：

| 规则候选状态 | AI 预判行为 |
| --- | --- |
| 无规则候选且低风险 | 默认不跑，除非人工手动触发 |
| P1 候选 | 批量任务可跑，用于补 root cause 和 recommended fix |
| P0 候选 | 优先跑，用于判断是否误报并补上下文解释 |
| `needs_evidence` | 只输出缺失证据，不能给高置信坏例结论 |
| 多规则命中同一链 | 综合所有候选，输出主根因和次要标签 |

约束与验收：

- AI 预判只能写 `suggested_labels`，不能写 `final_labels` 或 `review_decision`。
- AI 预判不能生成 regression draft 或 repair pack。
- context 缺失时，上下文类问题必须返回 `missing_evidence`。
- `repair-manual_triage-2ced95bdeea1` 这类问题必须能看到前文欠款/未付人工、trigger turn 和 result turn。
- 预判结果必须引用 `evidence_refs`，人工可采纳、修改或驳回。

## 11. Discovery Layer 与风险发现

> 本节只定义“问题如何被发现”。页面功能见 §10，规则候选见 §10.5，AI 预判见 §10.6，避免在发现层重复页面与标注细节。

### 11.1 目标

Discovery Layer 的目标不是替代人工审核，而是把海量 turn/session 收敛成可解释、可处理的问题候选。它必须回答三个问题：

1. 哪个 session / turn 可能有问题。
2. 为什么认为它有问题，证据来自规则、上下文还是 AI 预判。
3. 下一步应该人工采纳、驳回、补证据，还是生成修复包。

典型反例：`repair-manual_triage-2ced95bdeea1` 中“那把所有工人工资结了”被规则命中为高风险，但真实判断依赖前文是否已经列出工人、金额、欠款和支付上下文；仅看单 turn 会误判。

### 11.2 信号分工

| 信号 | 负责什么 | 不负责什么 | 输出 |
|---|---|---|---|
| 规则候选 | 快速发现明确模式，如禁用工人、工资结算、工具缺失、参数泄露 | 不直接给最终质量标签 | `rule_hit`、risk、evidence |
| 上下文分析 | 判断当前 turn 是否需要前后文、是否缺事件、是否缺工具链证据 | 不替代人工结论 | `context_scope`、missing evidence |
| AI 预判 | 结合规则候选和上下文给出建议标签、置信度、解释 | 不作为唯一真值 | `suggested_label`、confidence、reason |
| 人工审核 | 最终采纳、驳回、补证据、导出修复包 | 不做批量自动猜测 | final label、repair target |

### 11.3 风险评分

风险评分只用于排序，不等同于最终标签。

```text
risk_score = max(rule_score, ai_score) + context_bonus + severity_bonus - evidence_penalty
```

- `rule_score`：规则命中基础分。
- `ai_score`：AI 预判置信度。
- `context_bonus`：当前 turn 需要跨轮判断但上下文不足时加分。
- `severity_bonus`：涉及误执行、错工具、禁用工人、资金结算等高损场景时加分。
- `evidence_penalty`：证据链完整且模型行为合理时降分。

### 11.4 Context Analyzer MVP

| 能力 | 输入 | 产出 | 说明 |
|---|---|---|---|
| Session 聚合 | 同一 session 的 events.jsonl | timeline | 按 turn 还原用户输入、助手回复、工具、pending、事件 |
| 上下文窗口 | trigger turn + 前后 N 轮 | context window | AI 预判和人工审核默认看窗口，不只看单 turn |
| 证据完整性 | selected/actual tools、pending、event、router | evidence checklist | 标记 present / missing / needs_human |
| 问题链归并 | 多个相邻风险 turn | issue chain | 一个问题只进一个审核对象，避免重复标注 |

### 11.5 上线边界

| 边界 | 约束 |
|---|---|
| 真值来源 | 人工标签仍是最终真值；规则和 AI 只产生候选与建议 |
| 默认索引 | 以 session / issue chain 为主索引，turn 是证据节点，不再作为唯一工作入口 |
| 缺证据处理 | 缺事件文件、缺 router 决策、缺工具结果时必须显示缺失项，不能假装判断完成 |
| 成本控制 | AI 预判只对规则命中、高风险、人工点选或批量任务运行 |
| 可追溯 | 每个候选必须能回到原始 `events.jsonl`、request id、turn id 和证据片段 |

## 12. ReviewIssueChain 与人工审核口径

> 本节只定义审核对象和人工口径。每日质检页面如何展示见 §10.1，高级搜索边界见 §10.2。

### 12.1 核心对象

`ReviewIssueChain` 是飞轮审核的主对象：它把同一 session 中围绕同一风险点的一组 turn 聚合起来，避免审核员在“全部 turn”里孤立判断。

| 字段 | 作用 |
|---|---|
| `issue_chain_id` | 问题链唯一 ID |
| `session_id` | 所属会话，用于回看完整上下文 |
| `trigger_turn_id` | 触发候选的关键 turn |
| `context_turn_ids` | 需要一起查看的前后文 turn |
| `risk_level` | P0 / P1 / P2 排序用 |
| `candidate_sources` | `rule` / `ai` / `manual`，说明候选来源 |
| `evidence_checklist` | 事件、消息、工具、pending、router 等证据是否齐全 |
| `human_decision` | 人工判断：采纳、驳回、补证据、暂不处理 |
| `repair_target` | 是否进入修复包，以及修复目标 |

### 12.2 状态与动作

| 状态 | 含义 | 允许动作 |
|---|---|---|
| `candidate` | 系统发现候选，尚未人工处理 | 查看上下文、AI 预判、补证据、采纳、驳回 |
| `needs_evidence` | 证据不足，不能下结论 | 同步事件、选择缺失证据、暂存 |
| `accepted` | 人工确认是真问题 | 写修复目标、导出修复包 |
| `rejected` | 人工确认不是问题 | 保存驳回原因，用于规则/AI 反训 |
| `repair_ready` | 修复包字段完整 | 进入修复包页 |
| `evaluated` | 修复后已进入回归评测 | 查看指标、失败回流 |

### 12.3 标注员口径

- 先看 issue chain，不先看孤立 turn。
- 如果当前 turn 依赖前文，必须勾选 `context_turn_ids` 中的证据 turn。
- 缺工具、缺 pending、缺 router、缺事件文件时，优先标为 `needs_evidence`，不要硬判好坏。
- 规则候选命中不等于坏回复；必须结合助手回复、工具链和业务上下文判断。
- AI 预判只能作为建议，人工保存时必须写明采纳或驳回原因。

## 13. Evaluation 关键指标

| 指标 | 含义 | 目标 |
| --- | --- | --- |
| `pass_rate` | 通过率（replay 结果与 expected 一致） | ≥ 95% |
| `tool_selection_accuracy` | 工具选择准确率 | ≥ 90% |
| `pending_missed_rate` | pending 漏拦截率 | ≤ 2% |
| `hallucinated_execution_rate` | 幻觉执行率 | ≤ 1% |
| `parameter_mismatch_rate` | 工具参数/作用域与用户意图不一致率 | 持续下降，P0 样本 0 容忍 |
| `avg_token` | 平均 token 消耗 | 持续下降 |
| `avg_latency_p50` / `avg_latency_p95` | 延迟分位 | p95 ≤ 2s |
| `human_override_rate` | 人工干预率（采纳率 + 修改率） | ≤ 30% |
| `regression_ready_rate` | bad case 自动生成可审查回归草稿的比例 | ≥ 80% |
