# 07 — 可观测与运维

> 状态：草稿 | 维护：BlockShip | 关联：[01_Agent平台架构](./01_Agent平台架构.md)、[06_数据飞轮与评测](./06_数据飞轮与评测.md)

---

## 1. 为什么

2C4G 单机部署，但**不能因为小就忽略可观测**。每次写操作、每次工具调用、每条 LLM 请求都要能追溯：

- 用户说「上次给我记错了」→ 我能找到那条 trace
- 数据飞轮标注 bad case → 我能复制完整 debug JSON
- LLM 调用超时 → 我能看到是 Provider 问题还是 prompt 太长
- Token 成本飙升 → 我能定位到哪个 Skill 哪个用户

## 2. Trace 模型

每轮对话的 trace 结构：

```python
class TraceRecord:
    trace_id: str            # 全局唯一
    session_id: str          # 会话 ID
    turn_id: str             # 本轮 ID
    request_id: str          # HTTP 请求 ID
    farm_id: int             # 农场
    user_id: int             # 用户
    started_at: datetime
    ended_at: datetime

    # 链路段
    spans: list[Span]

class Span:
    span_id: str
    parent_id: str | None
    node: str                # entry / guardrail / planner / context_build / prompt_compose / llm_call / tool_call / reflect / response
    started_at: datetime
    ended_at: datetime
    duration_ms: int

    input_summary: dict      # 脱敏后摘要
    output_summary: dict
    error: dict | None

    # 关联段
    llm_meta: LLMMeta | None       # model / token / cost / cached
    tool_meta: ToolMeta | None     # skill_name / status / params_summary
    prompt_meta: PromptMeta | None # version / snippet_versions / token_before_budget
```

## 3. 存储分层

| 层 | 存储 | 用途 |
| --- | --- | --- |
| **热索引** | MySQL `trace_records` 表 | 列表查询：farm_id + started_at |
| **明细** | JSONL 文件 `logs/traces/{date}/{trace_id}.jsonl` | 详情页：按 trace_id 直接读 |
| **聚合** | MySQL `token_stats` / `skill_call_stats` | Dashboard：按天聚合 |

**禁止**：
- 热索引存全量 span（一行 trace 一行 MySQL）
- JSONL 列表扫描（必须先查 MySQL 索引拿文件路径）
- 同步阻塞主流程写 JSONL（必须 fire-and-forget 或 async）

## 4. 关键观测点

| 节点 | 观测内容 | 告警阈值 |
| --- | --- | --- |
| **request_entry** | 完整请求、用户、farm、session | — |
| **guardrail** | check_input / filter_output 命中规则 | 命中率 > 5% warning |
| **planner** | intent、candidates、tool_selector 决策 | 候选为空率 > 10% warning |
| **context_build** | selector 命中、token budget、压缩触发 | 超 budget 频率 > 20% warning |
| **prompt_compose** | snippet_versions、最终 token | — |
| **llm_call** | model、input/output tokens、latency、cached、cost | latency p95 > 5s critical；失败率 > 5% critical |
| **tool_call** | skill_name、status、duration、params_summary | 失败率 > 5% warning |
| **reflection** | checks 触发、reflection_trace 写入 | 触发率 > 10% warning |
| **pending_action** | 创建、确认、取消、过期 | 过期率 > 30% warning |
| **response_output** | 完整响应、SSE 事件数 | — |
| **memory_observe** | observation event 写入 | 失败 retry ≥ 3 critical |

## 6. 监控指标（Metrics）

### 6.1 请求级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `agent_request_total{status}` | counter | 每天调用量，status: success/fail/timeout |
| `agent_request_latency_seconds{quantile}` | histogram | 延迟分位 |
| `agent_request_concurrent` | gauge | 当前并发数 |

### 6.2 LLM 级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `llm_call_total{model, cached}` | counter | 大模型调用量 |
| `llm_latency_seconds{model}` | histogram | 大模型延迟 |
| `llm_tokens_total{direction, model}` | counter | Token 消耗（in/out） |
| `llm_cost_usd_total{model}` | counter | 美元成本 |
| `llm_error_total{model, reason}` | counter | 失败次数 |

### 6.3 业务级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `skill_call_total{skill_name, status}` | counter | 各 Skill 调用量 |
| `skill_latency_seconds{skill_name}` | histogram | Skill 延迟 |
| `pending_action_total{lifecycle}` | counter | created/confirmed/cancelled/expired |
| `reflection_trigger_total{check}` | counter | 各 check 触发次数 |
| `memory_observe_total{type}` | counter | observation 写入量 |
| `clarification_total` | counter | 澄清次数 |
| `tool_selection_empty_total` | counter | 候选工具为空次数 |

### 6.4 飞轮级

| 指标 | 类型 | 含义 |
| --- | --- | --- |
| `data_flywheel_sample_total{source}` | counter | 新样本数 |
| `data_flywheel_labeled_total{source}` | counter | 标注样本数 |
| `simulation_pass_rate` | gauge | 回归通过率 |
| `evaluation_pass_rate` | gauge | 评测通过率 |

## 7. 告警规则

| 告警 | 触发条件 | 级别 | 通知 |
| --- | --- | --- | --- |
| 高延迟 | P95 延迟 > 5s 持续 5 分钟 | warning | 站内信 |
| 错误率 | 错误率 > 1% 持续 5 分钟 | critical | 邮件 + 短信 |
| LLM 调用失败 | 5 分钟内失败 > 10 次 | critical | 邮件 + 短信 |
| Skill 失败率 | 失败率 > 5% | warning | 站内信 |
| Token 成本 | 日成本 > 预算 80% | warning | 站内信 |
| Token 成本 | 日成本 > 预算 100% | critical | 邮件 |
| 数据库连接池 | 使用率 > 80% | warning | 站内信 |
| 磁盘 | 使用率 > 80% | warning | 站内信 |
| 磁盘 | 使用率 > 95% | critical | 邮件 + 短信 |

通知通道：邮件（关键）/ 站内信（warning） / 短信（生产 critical）。

## 9. 日志规范

```python
logger = logging.getLogger(__name__)

logger.info(
    "agent_request_started",
    extra={
        "trace_id": trace_id,
        "session_id": session_id,
        "farm_id": farm_id,
        "intent": intent.value,
        "candidates": candidates,
    },
)
```

**禁止**：
- `print(...)` 调试
- `logger.debug` 写入生产（生产只 info 及以上）
- 日志带未脱敏的用户密码、token、个人信息
- 同步写入阻塞主流程

日志带 `trace_id` 是硬约束，CI 用 `scripts/check-log-format.sh` 验证。
