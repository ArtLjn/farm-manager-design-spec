## 1. 测试金字塔

```
                /\
               /  \         仿真 / 回归 / 评测
              /____\        ← 高价值、慢、人工评审
             /      \
            / 集成  \       API / Skill / Subgraph 集成
           /__________\     ← 中等数量、中速
          /            \
         /    单元测试   \    pytest / vitest / flutter_test
        /________________\  ← 数量最多、最快
```

| 层 | 工具 | 占比目标 | 运行时间 |
| --- | --- | --- | --- |
| 单元 | pytest / vitest / flutter_test | ≥ 70% | < 60s |
| 集成 | pytest + httpx / RTL / widget test | ≈ 20% | < 5min |
| 仿真 | `simulation/` 用例回放 | ≈ 5% | < 15min |
| 评测 | `evaluation/` 离线跑批 | ≈ 5% | 异步、按需 |

## 2. 各层职责

### 2.1 单元测试

- **范围**：纯函数、Service、Selector、Compressor、Prompt Renderer、Pydantic schema。
- **规则**：
  - 外部依赖必须 mock（DB、LLM、HTTP）。
  - 不 mock 被测对象本身。
  - 一个 test 函数只验证一个行为。
  - 命名：`test_<被测对象>_<场景>_<期望>`。
- **覆盖目标**：
  - `services/` ≥ 85%
  - `agent/runtime/` ≥ 80%
  - `skills/` ≥ 90%（每个 Skill 至少 3 类测试：Meta / Normal / Error）

### 2.2 集成测试

- **范围**：
  - API 层：HTTP 端到端，含认证、Farm 隔离、Pydantic 校验。
  - Skill 层：Skill → Service → DB（测试 SQLite）。
  - Subgraph：LangGraph 节点组合，不调真实 LLM。
- **规则**：
  - 用 `pytest-asyncio` + `httpx.AsyncClient`。
  - 测试 DB 用 SQLite in-memory 或独立 MySQL schema。
  - 每个 API 至少 3 个用例：正常 / 鉴权失败 / 越权。

### 2.3 仿真测试（Simulation）

详见 [03_Simulation仿真测试.md](./03_Simulation仿真测试.md)。

- **目的**：在真实 LLM 调用下回归 Agent 行为。
- **方法**：录制历史会话 → 改 Prompt / Skill 后回放 → 比对关键事件序列。
- **频率**：每次合并到 `main` 前必跑 smoke 集；每周跑 full 集。

### 2.4 评测（Evaluation）

详见 [04_Evaluation评测.md](./04_Evaluation评测.md)。

- **目的**：量化 Agent 质量（准确率、Tool 选择、Token 成本、延迟）。
- **方法**：固定测试集 → 离线跑批 → 生成 EvaluationReport。
- **频率**：每次发布前 + Prompt/Skill 大改后。

## 3. 测试环境

| 环境 | 用途 | DB | LLM |
| --- | --- | --- | --- |
| `local` | 开发自测 | SQLite | mock 或 staging key |
| `ci` | PR 检查 | SQLite in-memory | mock（不调真 LLM） |
| `staging` | 集成 / 仿真 | 独立 MySQL schema | 真 LLM（小配额） |
| `prod` | 监控 / A/B | 生产 MySQL | 生产 LLM |

## 4. CI 七道门

参考 [CLAUDE.md](../../CLAUDE.md) 可测试性分级。需求明确 + 可测试场景必须通过：

| 门 | 工具 | 失败处理 |
| --- | --- | --- |
| Lint | `ruff check . && ruff format --check .` | 阻塞 |
| 单元测试 | `pytest -q` | 阻塞 |
| 架构依赖 | `bash scripts/check-layer-deps.sh` | 阻塞 |
| Skill 契约 | `pytest tests/skills/test_skill_docs.py` | 阻塞 |
| Guide-Sensor 配对 | `bash scripts/check-guide-sensor-pairing.sh` | 阻塞（如适用） |
| 集成测试 | `pytest tests/api tests/agent` | 阻塞 |
| Simulation smoke | `python -m app.simulation.run --suite smoke` | 阻塞合并到 main |

## 5. 测试数据

### 5.1 Fixture

- 位于 `backend/tests/fixtures/`。
- 含：User / Farm / CropCycle / Cost / Conversation 等种子数据。
- 用 `conftest.py` 注入。

### 5.2 LLM 录制

- 用 `langsmith` 录制真实调用。
- 录制文件：`backend/tests/fixtures/llm_recordings/*.json`。
- 集成测试可 replay，避免真调 LLM。

### 5.3 仿真用例

- 位于 `backend/simulation/cases/`。
- 每个用例：`{user_input, expected_skills, expected_pending, expected_response_contains}`。
- 用 DataFlywheel 标注样本回灌。

## 6. 测试反模式（禁止）

- 测私有方法（应通过公共 API 测）。
- 一个 test 验证多个不相关行为。
- Mock 一切（外部依赖才 mock，内部协作别 mock）。
- 测试间共享状态（必须互相独立）。
- 只测 happy path。
- Snapshot 测试滥用（除非 UI / Prompt 这种稳定结构）。

## 7. 性能测试

- 后端：`locust` 压测关键端点（`/chat`, `/scenarios/parse`, `/data-flywheel/samples`）。
- 前端：Lighthouse CI（Admin Web）。
- 移动端：Flutter DevTools Performance。

性能基准（生产 2C4G）：

| 端点 | P95 延迟 | RPS |
| --- | --- | --- |
| `/chat`（含 LLM） | 8s | 5 |
| `/scenarios/parse` | 200ms | 50 |
| `/costs` CRUD | 100ms | 100 |

## 8. 安全测试

- 静态扫描：`gitleaks` / `detect-secrets`（密钥泄露）。
- 依赖扫描：`poetry audit` / `pnpm audit`。
- 越权测试：每个 API 至少一个 cross-farm 越权用例。
- Prompt Injection：`backend/tests/security/test_prompt_injection.py` 覆盖常见攻击模板。
